ATLASIP — BACKLOG PERFORMANCE v1.0
===========================================================
Document de référence à placer dans /doc/backlog_performance_v1.txt
Ce fichier regroupe tous les bugs, specs, correctifs et tâches liés aux PERFORMANCES d’AtlasIP.
Le parser v2 est exclu (géré dans un document séparé).

Structure : Criticité (P0 → P3) → Catégorie → Bug détaillé.


===========================================================
P0 — CRITICITÉ CATASTROPHIQUE (BLOQUANT)
===========================================================

P0-PERF-001
Catégorie : Backend / Réseau
Description : Pas de timeout global pour les requêtes RDAP/WHOIS/DNS.
Reproduction : Interroger une IP lente.
Observé : L’application reste bloquée indéfiniment.
Attendu : Timeout global (ex : 5s) + fallback propre.
Impact : Analyse impossible.
Sprint : Performance v1.

P0-PERF-002
Catégorie : Backend / Réseau
Description : Pas de gestion centralisée des erreurs réseau.
Reproduction : Couper la connexion.
Observé : Panique ou erreur non catchée.
Attendu : Gestion d’erreur propre + message clair.
Impact : Crash potentiel.
Sprint : Performance v1.

P0-PERF-003
Catégorie : Backend / Parallélisation
Description : Les requêtes RDAP sont exécutées séquentiellement.
Reproduction : Analyser 10 IPs.
Observé : Temps x10.
Attendu : Parallélisation (tokio join_all).
Impact : Lenteur critique.
Sprint : Performance v1.

P0-PERF-004
Catégorie : Backend / Caching
Description : Aucun caching local n’est implémenté.
Reproduction : Analyser 8.8.8.8 deux fois.
Observé : Deux requêtes RDAP identiques.
Attendu : Cache TTL (ex : 1h).
Impact : Charge inutile + lenteur.
Sprint : Performance v1.

SPÉCIFICATIONS PERFORMANCE (HORS PARSER)
===========================================================

1. Parallélisation
- RDAP en parallèle
- WHOIS en parallèle
- DNS en parallèle
- join_all obligatoire

2. Caching
- cache local (HashMap + TTL)
- clé = IP ou hostname
- TTL configurable (1h par défaut)
- invalidation propre

3. Batching
- DNS batch
- WHOIS batch (optionnel)
- RDAP batch (si possible)

4. Timeouts
- timeout global (5s)
- timeout DNS (2s)
- timeout WHOIS (3s)
- timeout RDAP (3s)

5. Retry intelligent
- retry exponentiel
- max 2 tentatives
- logs en cas d’échec

6. Logs structurés
- format JSON
- niveau (info, warn, error)
- latence par requête

7. Metrics
- latence RDAP
- latence WHOIS
- latence DNS
- taux d’erreur
- taux de fallback

8. Frontend
- virtualisation DataGrid
- réduction des re-renders
- optimisation autosize
- optimisation layout
