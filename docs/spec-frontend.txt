# AtlasIP — Frontend Specification (v1.0)
Author: Absol3m
Language: English
Scope: Frontend architecture, UX rules, layout, components, and interactions.

--------------------------------------------------
1. Overview
--------------------------------------------------
AtlasIP’s frontend is a modern, responsive, desktop‑first interface built to
provide a clean, efficient workflow for IP OSINT analysis. It interacts with
the local backend API and presents results in a structured, intuitive way.

The frontend must:
- Be simple, fast, and predictable.
- Follow modern UX patterns (2026 standards).
- Use a left sidebar + main workspace layout.
- Provide a unified analysis workflow (input + results on the same page).
- Offer a detailed IP view with structured sections.
- Support internationalization (English/French).
- Persist user preferences (theme, column visibility, etc.).
- Be fully compatible with a future Tauri desktop build.

--------------------------------------------------
2. Global Layout
--------------------------------------------------
The application uses a two‑panel layout:

Left Sidebar (fixed)
Main Workspace (dynamic)

### 2.1. Sidebar
Always visible. Contains:
- AtlasIP logo + name
- Navigation:
  - Analysis
  - History (future)
  - Settings
  - About
- Theme switch (light/dark)
- Version number (bottom)

Rules:
- Width: fixed (240–260px)
- Never collapses automatically
- No nested menus

### 2.2. Main Workspace
Displays the active page:
- Analysis page (default)
- IP detail page
- Settings
- About

Must support:
- Scrollable content
- Sticky header elements
- Responsive resizing

--------------------------------------------------
3. UX Principles
--------------------------------------------------
The frontend must follow these principles:

### 3.1. Clarity
- No ambiguous labels
- No hidden actions
- No modal-heavy interactions

### 3.2. Speed
- Minimal clicks
- No unnecessary navigation
- Input always visible on the analysis page

### 3.3. Predictability
- Same layout across pages
- Consistent component behavior
- No surprise state resets

### 3.4. Professional OSINT Tooling
- Dense information, but readable
- Keyboard-friendly
- Export-friendly

--------------------------------------------------
4. Navigation Rules
--------------------------------------------------
- Navigation is client-side (SPA).
- Sidebar selection highlights the current page.
- Navigating to an IP detail page uses route: `/ip/<ip>`.
- Returning to analysis preserves:
  - Input content
  - Table state
  - Scroll position

--------------------------------------------------
5. Components
--------------------------------------------------

### 5.1. InputField (Analysis Page)
A smart input component supporting:
- Single IP
- Multiple IPs (one per line)
- Pasted email headers (auto-extraction)
- Pasted text blocks (auto-detection)
- Auto-cleaning (trim, dedupe, normalize)

Placeholders:
- Empty: “Enter one or more IP addresses (one per line)”
- After header detection: “IPs extracted from email header”
- After text detection: “IPs extracted from text”

### 5.2. DataGrid (Results Table)
Defined in detail in `spec-datagrid.txt`.

### 5.3. IP Detail Sections
Used in `/ip/<ip>` page:
- Summary
- Organization
- Network
- Contacts
- DNS
- WHOIS (raw)
- Metadata

### 5.4. Settings Components
- Toggle switches
- Dropdowns
- Text inputs
- Proxy configuration fields

--------------------------------------------------
6. Internationalization
--------------------------------------------------
Frontend loads:
- `/locales/en.json`
- `/locales/fr.json`

Rules:
- English is default
- Missing keys fallback to English
- No hardcoded strings in components

--------------------------------------------------
7. Error Handling (Frontend)
--------------------------------------------------
Errors must be:
- Non-blocking
- Displayed in a toast or inline message
- Never shown as raw backend errors

Examples:
- “Invalid IP address”
- “Private or reserved IPs are ignored”
- “Lookup failed (rate limit or network error)”

--------------------------------------------------
8. Themes & Styling
--------------------------------------------------
Two themes:
- Light
- Dark

Rules:
- Use CSS variables for colors
- High contrast for table data
- No low-opacity text
- Icons must adapt to theme

--------------------------------------------------
9. Performance Requirements
--------------------------------------------------
- Must handle 500+ IPs without UI freeze
- Table virtualization required
- No blocking loops
- Debounced input parsing

--------------------------------------------------
10. Future Extensions
--------------------------------------------------
- Browser extension integration
- Drag & drop file import
- Saved IP lists
- Session history
