# AtlasIP — Analysis Page Specification (v1.0)
Author: Absol3m
Language: English
Scope: Full UX and functional specification for the main IP analysis page.

--------------------------------------------------
1. Overview
--------------------------------------------------
The Analysis Page is the central workspace of AtlasIP. It combines:
- An intelligent input field for IPs, hostnames, and email headers
- A results DataGrid
- A unified workflow for bulk analysis

The page must remain simple, fast, and predictable, even with large datasets.

--------------------------------------------------
2. Layout Structure
--------------------------------------------------
The page contains three main sections:

1. Sticky Header
2. Input Area
3. Results Area (DataGrid)

### 2.1. Sticky Header
Always visible at the top of the workspace.

Contains:
- Page title: “IP Analysis”
- Optional quick actions (future):
  - Clear all
  - Import file
  - Export results

Rules:
- Does not scroll with content
- Height: 60–70px
- No heavy UI elements

### 2.2. Input Area
Located directly under the sticky header.

Contains:
- Smart multi-mode input field
- “Analyze” button
- Optional parsing feedback

Rules:
- Always visible (sticky)
- Expands vertically when multiline
- Must not overlap the DataGrid

### 2.3. Results Area
Displays the DataGrid (see spec-datagrid.txt).

Rules:
- Fills remaining vertical space
- Scrollable
- Must not shift the input area

--------------------------------------------------
3. Input Field Specification
--------------------------------------------------
The input field is a core component with multiple modes.

### 3.1. Supported Inputs
- Single IP
- Multiple IPs (newline-separated)
- Hostnames (resolved before lookup)
- Email headers (auto-extraction)
- Arbitrary text containing IPs

### 3.2. Behavior
- Auto-detects IPs on paste
- Cleans input (trim, dedupe, normalize)
- Filters private/reserved IPs
- Displays parsing feedback

### 3.3. Placeholders
Depending on context:

Default:
“Enter one or more IP addresses (one per line)”

After header detection:
“IPs extracted from email header”

After text detection:
“IPs extracted from text”

After hostname resolution:
“Hostname resolved to IP(s)”

### 3.4. Auto-Extraction Rules
When user pastes content:
- If it matches email header patterns → header mode
- If it contains multiple IPs → bulk mode
- If it contains a hostname → resolve automatically
- If no IP found → show inline error

### 3.5. Validation Rules
Each IP must be:
- Valid IPv4 or IPv6
- Public (backend rules)
- Not duplicated

Invalid entries:
- Highlighted in red
- Tooltip explaining the issue

--------------------------------------------------
4. Analyze Button
--------------------------------------------------
Triggers a bulk lookup via:
POST /lookup/bulk

### 4.1. States
- Enabled (default)
- Disabled (no valid IPs)
- Loading (spinner)
- Success (green flash)
- Error (red flash)

### 4.2. Behavior
- Sends cleaned list of IPs
- Does not block UI
- Scrolls to DataGrid on success

--------------------------------------------------
5. Results Area (DataGrid Integration)
--------------------------------------------------
The DataGrid displays the lookup results.

### 5.1. Initial State
If no results:
Centered placeholder:
“No results yet. Enter IPs above to begin.”

### 5.2. After Lookup
- Rows appear immediately
- Table scroll position resets to top
- Errors appear inline per row

### 5.3. Editing from the Table
Editing the IP column:
- Updates the input field automatically
- Marks row as “pending”
- Allows re-analysis

--------------------------------------------------
6. Workflow Scenarios
--------------------------------------------------

### 6.1. Scenario A — Single IP
User enters:
`8.8.8.8`
→ Press Analyze  
→ Table shows one row  
→ User can click row to open detail page

### 6.2. Scenario B — Multiple IPs
User pastes:
8.8.8.8
1.1.1.1
→ Input auto-expands  
→ Analyze  
→ Table shows two rows

### 6.3. Scenario C — Email Header
User pastes a full header:
→ Auto-detect  
→ Extract IPs  
→ Replace input with cleaned list  
→ Analyze

### 6.4. Scenario D — Arbitrary Text
User pastes:
“Suspicious login from 185.220.101.4 at 03:21 UTC”
→ Extract IP  
→ Analyze

### 6.5. Scenario E — Editing in Table
User edits IP in table:
→ Input updates  
→ Row marked pending  
→ User clicks Analyze again

--------------------------------------------------
7. Error Handling
--------------------------------------------------

### 7.1. Input Errors
Displayed inline:
- Invalid IP
- Private/reserved IP
- No IP found

### 7.2. Lookup Errors
Displayed in table row:
- Rate limit
- Network error
- WHOIS access denied
- RDAP unavailable

### 7.3. Global Errors
Displayed as toast:
- Backend unreachable
- Export failure

--------------------------------------------------
8. Performance Requirements
--------------------------------------------------
- Input parsing must be debounced (50–100ms)
- Must handle 500+ IPs without lag
- No blocking UI during lookup
- Table virtualization required

--------------------------------------------------
9. Future Enhancements
--------------------------------------------------
- Drag & drop file import
- Saved IP lists
- Auto-analysis on paste
- Inline WHOIS preview
