# AtlasIP — IP Detail Page Specification (v1.0)
Author: Absol3m
Language: English
Scope: Full UX and functional specification for the IP detail page (/ip/<ip>).

--------------------------------------------------
1. Overview
--------------------------------------------------
The IP Detail Page provides a structured, human-readable view of all information
collected for a single IP address. It is designed for analysts who need to
quickly understand ownership, network ranges, abuse contacts, and DNS/WHOIS
details.

The page must:
- Be clean and information-dense
- Use clear sections with consistent formatting
- Support deep linking (/ip/<ip>)
- Load instantly when navigated from the DataGrid
- Allow exporting the detailed view (HTML/TXT)
- Support dark/light themes

--------------------------------------------------
2. Layout Structure
--------------------------------------------------
The page is composed of:

1. Sticky Header
2. Summary Card
3. Sectioned Detail View
4. Raw WHOIS Panel
5. Metadata Footer

### 2.1. Sticky Header
Always visible at the top.

Contains:
- Title: “IP Details — <ip>”
- Back button (“← Back to Analysis”)
- Export dropdown (HTML, TXT, raw WHOIS)
- Refresh button (re-run lookup)

Rules:
- Height: 60–70px
- No heavy UI elements
- Must not scroll with content

### 2.2. Summary Card
A compact overview of the most important fields.

Fields:
- IP address
- Hostname (if any)
- Country (flag + code)
- Organization / Owner
- Netname
- CIDR
- Range (start → end)

Rules:
- Displayed as a card
- High contrast
- Icons for country, org, DNS

--------------------------------------------------
3. Sections
--------------------------------------------------
The detail page is divided into collapsible sections.

All sections default to expanded.

### 3.1. Organization
Fields:
- Owner name
- Organization name
- Netname
- RIR (ARIN, RIPE, APNIC, LACNIC, AFRINIC)
- WHOIS source

### 3.2. Network
Fields:
- CIDR
- Range start
- Range end
- Parent network (if available)
- Registration date
- Last updated

### 3.3. Contacts
Fields:
- Abuse emails
- Technical emails
- Phone numbers
- Fax numbers
- Postal address

Rules:
- Emails displayed as clickable mailto:
- Multiple emails separated by “; ”

### 3.4. DNS Information
Fields:
- PTR (reverse DNS)
- Resolved hostname
- Additional DNS notes (future)

Rules:
- PTR lookup failures displayed as “No PTR record”

### 3.5. Geolocation (Optional / Future)
Fields:
- Country
- Region
- City
- Coordinates
- “Open in Google Maps” button

### 3.6. Raw WHOIS
Displayed in a collapsible panel.

Rules:
- Monospace font
- Preserves formatting
- Scrollable
- Copy-to-clipboard button

--------------------------------------------------
4. Loading & Error States
--------------------------------------------------

### 4.1. Loading State
Displayed while fetching:
- Skeleton placeholders for each section
- Spinner in header

### 4.2. Error State
If lookup fails:
- Large error message
- “Retry” button
- Details (rate limit, network error, etc.)

--------------------------------------------------
5. Navigation Rules
--------------------------------------------------
- Navigating from DataGrid → opens `/ip/<ip>`
- Back button returns to analysis page
- Analysis page state must be preserved:
  - Input content
  - Table state
  - Scroll position

Direct navigation to `/ip/<ip>`:
- Performs a fresh lookup
- Shows loading state

--------------------------------------------------
6. Export Behavior
--------------------------------------------------
Exports are generated by backend via:
GET /export?format=...

Formats:
- HTML (detailed view)
- TXT (vertical)
- Raw WHOIS

Rules:
- Export uses backend data, not frontend state
- Hidden fields are still exported
- Export does not include UI elements

--------------------------------------------------
7. Internationalization
--------------------------------------------------
All labels must use i18n keys:
- “Organization”
- “Network”
- “Contacts”
- “Raw WHOIS”
- “Abuse Email”
- etc.

Missing translations fallback to English.

--------------------------------------------------
8. Performance Requirements
--------------------------------------------------
- Page must load in <150ms after data is available
- Raw WHOIS panel must not block rendering
- No layout shifts during loading

--------------------------------------------------
9. Future Enhancements
--------------------------------------------------
- ASN lookup
- Threat intelligence integration
- IP reputation scoring
- Timeline of ownership changes
- WHOIS diff between lookups
